01 — Foundations

YunoBall Frontend Specification

YunoBall is a search-first NFL analytics product: ask a question in plain language, get an answer computed from a real historical warehouse — narration, a chart, a table, and the exact SQL behind it. This document is the single source of truth for how that interface looks, behaves, and gets built.

The design language is a deliberate blend: the search-first immediacy of StatsMuse, the restraint and speed of Linear, the typographic calm of Vercel and Apple, and the information density of ESPN and The Athletic. We borrow these philosophies — we do not copy any of their brands. YunoBall's identity is its own: a dark neutral canvas, a single emerald accent used sparingly, depth from background contrast instead of borders, and near-zero chrome. When the active context is one team, the accent re-tints to that team's branding.

Revision note (2026-07 UI refinement): the product moved from the original light/indigo concept to a dark, borderless system with an emerald brand accent and dynamic team theming. The palette, radii and elevation below reflect the shipped system; the runtime source of truth for every token is apps/web/app/globals.css. Rendered examples in this document still use the doc's own light chrome — trust the token tables, not the demos.

What the product does

  • Search (/) — a hero prompt, an autofocused input, and example chips. One query returns one answer card.
  • Answer card — AI narration, resolved entity chips, an optional bar chart, a data table, and a collapsible "Show the query" SQL block. Every answer is shareable.
  • Leaderboards (/leaderboards) — season-scoped leader boards in a responsive grid, powered by the same warehouse.
  • Shareable answers (/a/[id]) — a permalinked, read-only render of any answer card.
North star: the interface should feel instant and trustworthy. Trust comes from showing the work (data + SQL); speed comes from a lightweight UI with no decorative weight. When in doubt, remove.
02 — Foundations

Design principles

Seven rules that resolve most design decisions before they reach a debate.

Search-first
The query box is the product's center of gravity. It is the largest interactive element on the home screen and stays reachable (sticky) everywhere else.
Content over chrome
Data is the hero. Depth comes from the surface ramp (background contrast) and one soft shadow — borders are rare by design, reserved for hairline dividers inside dense data. If removing a border makes the interface cleaner, remove it.
Dense but readable
Pack information tightly — compact tables, tabular numerals, tight vertical rhythm — while protecting legibility with generous line-height in prose and clear hierarchy.
Fast & lightweight
Perceived speed is a feature. Skeletons over spinners, optimistic transitions under 240ms, no blocking animation, minimal payload.
Typography carries it
With almost no ornament, type does the heavy lifting: weight, size, letter-spacing, and color establish hierarchy. Numbers use tabular figures.
Consistent & predictable
Every spacing value, radius, and color comes from a token. Same interaction, same feedback, everywhere.
Accessible by default
Keyboard-complete, visible focus, AA contrast, honored prefers-reduced-motion. Accessibility is a baseline, not a pass.
DoLet whitespace, typography and surface contrast separate sections. Use the accent once per view to point at the primary action or datum.
Don'tStack bordered cards inside bordered cards, gradients, neon colors, or animated flourishes. No dashboard clutter, no decorative UI that doesn't improve usability.

Color system

A tight, semantic palette on a dark neutral canvas. The surface ramp does the structural work — depth is background contrast, not lines. Emerald is the sole brand accent and appears sparingly: search focus, selected states, primary actions, links, positive metrics. Neutral dominates everywhere else.

Contrast guardrails: muted text (#a1a1aa, ≥5.6:1) and faint text (#95959f, ≥4.8:1) both clear WCAG AA on every surface in the ramp, including the hover tier — the worst case for any text tone is faint on --bg-hover. Reserve faint for tertiary labels. Accent-colored text uses --accent-ink #34d399; solid accent fills use --accent with --accent-contrast ink on top.

Dynamic team theming

When the active context is exactly one team — a team page, a player profile, or an answer whose entities resolve to a single team — the accent token family (--accent, --accent-hover, --accent-ink, --accent-soft, --accent-border, --accent-contrast, --chart) is overridden on that container with the team's brand hue via scoped CSS variables (apps/web/app/lib/teamTheme.ts). Only accent-driven UI re-tints — buttons, active tabs, links, chart marks, stat highlights. Backgrounds and neutrals never recolor, and ambiguous contexts (head-to-heads, multi-team answers) stay emerald. Hues are static brand-adjacent tones tuned for the dark surface, with fill ink chosen by computed contrast.

Typography

Body family Geist (fallback SF Pro); display family Barlow Condensed for headlines, scores and big numbers — uppercase, heavy, sports-page energy. Tabular numerals for all metrics. Hierarchy comes from type (size, weight, tone), not from boxes or rules. One scale, used everywhere:

03 — Foundations

Page layout

A centered single-column reading measure, widened only where data needs room.

Reading column
max-width: 720px — home hero, a single answer, share pages. Keeps line length and focus tight.
Content column
max-width: 980px — leaderboards and any multi-column data grid.
Gutters
20px on mobile, 24px on desktop. Content never touches the viewport edge.
Vertical rhythm
Hero pads 72px top; sections separate with a single hairline + --s-10 (40px). Bottom pad 120px so the last answer clears the fold.
Grid
Leaderboards use repeat(auto-fill, minmax(420px, 1fr)) — reflows 1→2→3 columns with no breakpoints to maintain. Prefer intrinsic (auto-fit/fill) grids over fixed media-query columns.
One primary column, always centered. There is no persistent app sidebar in the product — navigation is a thin sticky top bar. The only sidebar in this system is a docs/reference TOC (like this page), never the main app shell.

04 — Experience

Search experience

The single most important interaction. It must feel weightless and always within reach.

Search input — default → focus
↵

Behavior

  • Autofocus on load of the home page — the user can type immediately.
  • Sticky on results & deeper pages — the query box docks below the nav so a follow-up is always one keystroke away (StatsMuse-style persistence).
  • Global shortcut — / or ⌘K focuses the input from anywhere; Esc clears/blurs.
  • Submit on Enter; trims whitespace; ignores empty queries.
  • Example chips below the input seed the query and run it immediately — teaching the range of questions and removing the blank-page problem.
  • One answer per query. Submitting resets prior result/error and shows a skeleton (see §10) — never stack results.

Interaction states

Default
Borderless Spotlight-style bar: --elevated surface, --shadow-card, faint placeholder. The largest interactive element on the page.
Hover
Surface brightens to --bg-soft; cursor text.
Focus
2px --accent-border ring + --shadow-pop lift. No outline suppression without a replacement ring.
Loading
Input stays enabled; an AnswerSkeleton renders below. Optionally a right-aligned inline spinner glyph.
Disabled
Only during hard failures; reduce opacity, keep text legible.
A11y: input has a visible or aria-label label; chips are real <button>s; the active chip carries aria-pressed; the results region is aria-live="polite" so screen readers announce the answer.

Card anatomy (top → bottom)

  1. Narration — 24px/700, the plain-language answer. Leads the card; readable as a standalone sentence.
  2. Entity chips — resolved players/teams/seasons as static stat-chips; each carries a title with type + confidence.
  3. Chart — an optional horizontal bar chart, shown only when the data has a labeled dimension and a numeric column that actually varies (see §7).
  4. Table — the full result set, horizontally scrollable on overflow (see §6).
  5. Footer actions — "Show the query" (toggles the SQL block), "Share" (copies /a/[id] permalink, confirms inline), and a subtle cached badge when served from cache.
  6. SQL block — collapsed by default; monospace on --bg-soft; horizontally scrollable. This is the trust primitive — every answer shows its work.
Spacing
Card padding 22px; --s-3–--s-4 between internal blocks; footer separated by a hairline + --s-3.
Enter animation
Fade + 4px rise over 240ms, once. No looping motion. Respects reduced-motion.
Share pages
/a/[id] renders the identical card read-only (no live search box above it), server-fetched for shareable/OG-friendly output.

Compact, scannable, keyboard-friendly. Numbers right-aligned and tabular; structure from hairlines, not boxes.

Sticky header
Header row pins to the top of a scrollable table body so column context never scrolls away.
Sorting
Click a header to sort; a ▲/▼ arrow in the accent color marks the active column and direction. Headers are focusable and toggle on Enter/Space; carry aria-sort.
Filtering
Above-table filter pills (see §9) narrow rows; active filters are removable and announced. Empty result → the empty state (§11).
Compact spacing
10–11px vertical padding — dense but tappable. Do not inflate row height for "breathing room"; density is the point.
Zebra striping
Even rows on --bg-soft for horizontal tracking across wide tables.
Hover
Whole row tints to --accent-soft to anchor the eye; cursor default (or pointer if the row is expandable/navigable).
Numeric columns
Right-aligned, tabular figures, thousands separators. Text columns left-aligned. First column semibold as the row's identity.
Keyboard nav
Arrow-key row/cell traversal for large tables; Enter expands a row; focus ring visible on the focused cell.
Responsive overflow
Wrap in a horizontally scrollable container with a rounded border; never let a wide table break the page. Consider hiding low-priority columns below 640px (see §13).
Expandable rows
When a row has detail (game log, splits), a chevron toggles an inset sub-panel; the trigger uses aria-expanded.
DoRight-align and tabularize every number; keep headers uppercase-caption and muted; separate rows with one hairline.
Don'tAdd vertical gridlines, heavy borders, alternating bright fills, or per-cell padding large enough to halve the visible row count.

Charts & visualizations

Charts summarize; the table is the source of truth. Default to the horizontal bar — honest, sortable, label-friendly.

Horizontal bar chart — ranked, single accent
Patrick Mahomes5,250
Justin Herbert4,739
Joe Burrow4,475
Josh Allen4,283
Jalen Hurts3,701
When to chart
Only when there's a labeled dimension and a numeric column whose values actually vary. A constant column (e.g. every row is season=2023) is never charted — a flat bar chart is noise.
Chart type
Horizontal bars for rankings/leaders (default). Line for trends over time; grouped/overlaid bars for comparisons of ≤3 entities.
Color
One accent for a single series; a small, fixed, colorblind-safe categorical set (accent, teal, amber, slate) for comparisons — assigned consistently so an entity keeps its color across views.
Bar count
Cap at ~15 bars; beyond that, truncate and defer to the table. Long labels ellipsize.
Axes
Value labels sit at the bar end (no separate axis needed for short lists). For line/time charts, label axes with sparse, readable ticks; avoid gridline clutter.
Transitions
Bars grow from 0 → value on mount over ~240ms; sort/filter re-animates widths smoothly. All motion respects reduced-motion.
Tooltips
Hover/focus a bar or point → a compact tooltip with the label + exact value. Keyboard-reachable; dismiss on blur/Esc.
Comparison & zoom
Comparison views overlay entities with a legend; time charts support brush-to-zoom and range filtering. Keep controls minimal and out of the way until hovered.
Accessibility
Chart container has role="img" + descriptive aria-label; the underlying table is always present as the accessible, machine-readable equivalent.

Navigation

One thin, sticky, translucent top bar. Wordmark left, primary links right. Nothing more.

YunoBall Search Leaderboards
Structure
Wordmark (YunoBall, links home) left; text links (Search, Leaderboards) right. 14–22px padding.
Sticky + blur
position: sticky; top: 0 with an 85%-opacity white background and backdrop-filter: blur(8px) so content scrolls beneath, legibly. z-index: 10.
Active state
Current route link uses --accent; others --muted, darkening to --text on hover. Mark active with aria-current="page".
Mobile
The two links fit inline down to small screens — no hamburger needed. If nav grows, collapse to a menu button (44px target) opening a sheet, not a full overlay.
Sub-navigation
Handled inline per page (e.g. the leaderboards season <select>), never as a persistent second bar.
A11y: a "Skip to content" link precedes the nav for keyboard users; links are real anchors; focus order is left→right.

10 — System

Loading states

Never a bare spinner on a blank page. Show the shape of what's coming.

Answer skeleton — mirrors the answer card
  • Skeletons match layout — the answer skeleton mirrors narration → chart bars → table rows; the leaderboard skeleton mirrors a board's ranked bars.
  • Instant feedback — render the skeleton the moment a query submits; keep the input interactive.
  • No layout shift — skeleton dimensions approximate the real content so the card doesn't jump when data arrives.
  • Cached answers skip the skeleton (they're instant) and show the cached badge.
  • Reduced motion — shimmer animation is disabled; a static grey block remains.
11 — System

Empty states

A dead end should teach the next move, not apologize.

No results
🔍

No data for that question yet

We couldn't find stats matching your query. Try a different season, player, or phrasing.

  • Contexts — no search results, an empty leaderboard board (hidden entirely if zero rows), a filtered table with no matches, a fresh page before first query.
  • Anatomy — quiet glyph, one-line headline, a short explanation, and a concrete next action (example chips, "clear filters", or a link).
  • Tone — helpful and specific ("no data for that question"), never a generic "Nothing here."
  • Style — centered, dashed border on --bg-soft, muted text. Low visual weight — it's a state, not a feature.
12 — System

Error states

Honest, calm, and recoverable. Say what happened and offer a way forward.

Request failure
⚠️

Something went wrong

We couldn't reach the stats service (error 503). Your question is fine — this is on us.

  • Inline errors — a request failure renders a danger-tinted message directly below the search input (not a modal), preserving the query so it can be retried or edited.
  • Levels — field validation (inline, subtle) · request failure (card-level, with retry) · fatal/route error (full-page boundary with a home link).
  • Content — plain-language cause, a status code when useful, and a primary "Try again" action. Never dump a raw stack trace to users.
  • Color — --danger text/border on --danger-soft; reserve red exclusively for errors, never for emphasis.
  • A11y — error container is role="alert" so it's announced immediately; focus moves to it or its retry action.

Mobile behavior

The same product, reflowed — not a stripped-down variant. Search stays front and center.

Breakpoints
≤560px mobile · 561–900px tablet · >900px desktop. Prefer intrinsic grids over hard breakpoints where possible.
Gutters & type
Gutters shrink to 16–20px; the hero display scales from ~52px → ~32px; body size holds at 16px for readability.
Search
Full-bleed input (minus gutters), still autofocused; sticky search remains one tap away. Example chips wrap and scroll.
Tables
Horizontal scroll within the rounded container; low-priority columns hide below 640px; identity + primary metric always visible. Tap a row to expand detail.
Charts
Bars go full-width; labels shorten/ellipsize; value labels stay at bar end.
Grids

Targets
All interactive elements ≥44×44px; increased spacing between tappable chips/links.

14 — System

Accessibility requirements

Non-negotiable baseline: WCAG 2.1 AA. Accessibility ships with every component, not after.

Contrast
Body/critical text ≥ 4.5:1; large text ≥ 3:1. Muted is for secondary text, faint for tertiary labels; both clear 4.5:1 on every surface in the ramp (see the color-system guardrails). Accent-colored text always uses --accent-ink; solid accent fills pair with --accent-contrast.
Focus
Every interactive element shows a visible :focus-visible ring (2px accent, 2px offset). Never remove outlines without a replacement. Logical DOM/tab order.
Keyboard
Fully operable without a mouse: / ⁄ ⌘K focus search, table sort/traverse, dropdown arrow-nav + type-ahead, modal focus-trap + Esc, tooltip on focus.
Semantics
Real elements: <button>, <table>/<th scope>, <nav>, <main>, headings in order. ARIA only to fill gaps (aria-sort, aria-live, aria-current, role="dialog").
Announcements
Results region aria-live="polite"; loading aria-busy; errors role="alert".
Color independence
Never encode meaning in color alone — deltas add ▲/▼ + words, chart series get labels/patterns, sort adds an arrow glyph.
Motion
Honor prefers-reduced-motion: disable shimmer, bar-grow, and entrance transitions; keep instant state changes.
Skip link
A "Skip to content" link is the first focusable element.
Targets
≥44×44px on touch; adequate spacing between adjacent controls.

  • Search — autofocus; / ⁄ ⌘K focus; borderless elevated bar with an accent focus ring; results in aria-live.
  • AnswerCard — borderless Surface; interpretation strip as an inset well; accent family re-tints via teamTheme() when one team is in context.
  • Tables — SortTable over .yb-table: no wrapper frame, sticky header on a stronger hairline, row hover on --elevated, numeric right-alignment with tabular figures.
  • Lists — leaders, performers, boards are spacing + typography rows with surface hovers, not bordered cards.
  • Skeleton — every async region has a matching skeleton with a 150ms anti-flash delay and aria-busy.
  • States — shared empty/error state styles (.yb-state), route-level error.tsx boundary.
Guiding constraint: adopt libraries for behavior and accessibility, not for their default aesthetics. Every third-party component gets restyled to these tokens so the product reads as one coherent, YunoBall-branded surface — fast, quiet, and data-first.
